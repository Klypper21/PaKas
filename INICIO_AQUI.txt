QUÉ HACE EL SISTEMA
════════════════════════════════════════════════════════════════════════════════

ANTES (❌ Sistema antiguo):
  └─ 1 Producto = 1 Stock + 1 Precio
     └─ Color/Talla = Solo texto (sin validación)
     └─ Posible vender más de lo disponible (overselling)

DESPUÉS (✅ Sistema moderno tipo Amazon):
  └─ 1 Producto Base = Múltiples Variaciones
     ├─ Variación 1: Color=Rojo, Talla=S   → SKU=TSHIRT-RED-S, Stock=15, Precio=25
     ├─ Variación 2: Color=Rojo, Talla=M   → SKU=TSHIRT-RED-M, Stock=20, Precio=25
     ├─ Variación 3: Color=Azul, Talla=S   → SKU=TSHIRT-BLUE-S, Stock=8, Precio=25
     ├─ Variación 4: Color=Azul, Talla=M   → SKU=TSHIRT-BLUE-M, Stock=0, Precio=25 (AGOTADO)
     └─ Variación 5: Color=Negro, Talla=S  → SKU=TSHIRT-BLACK-S, Stock=5, Precio=28 (Precio diferente)

TROUBLESHOOTING RÁPIDO
════════════════════════════════════════════════════════════════════════════════

❓ "Variaciones no aparecen"
→ ¿Ejecutaste sql/product_variations.sql en Supabase?
→ ¿Insertaste datos en product_variations con parent_product_id correcto?

❓ "Selectores están vacíos"
→ Abre DevTools (F12) → Console
→ Escribe: Variations.loadVariations(1)
→ Si retorna [], no hay datos para ese producto

❓ "El precio no cambia"
→ Verifica que la variación tiene un 'price' especificado
→ Si 'price' es NULL, usa el precio del producto base

❓ "Stock incorrecto"
→ En Supabase: SELECT * FROM product_variations WHERE parent_product_id = 1
→ Verifica que 'stock' tenga el valor correcto

→ Ver CHECKLIST_IMPLEMENTACION.md para más soluciones
